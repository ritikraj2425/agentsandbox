# AgentSandbox System Architecture

This document outlines the internal design, component structure, execution lifecycles, and data flows of **AgentSandbox**.

---

## 1. High-Level Architecture Overview

AgentSandbox is a secure execution layer designed to run untrusted shell commands and actions generated by AI agents. It uses a decoupled modular structure, ensuring that the CLI wrapper, policy evaluation, tracing/logging, and the sandboxing runtime backends are completely independent.

```
                          +------------------+
                          |   CLI Wrapper    |  (cmd/agentsandbox/main.go)
                          +--------+---------+
                                   |
                                   v
                          +------------------+
                          |  Action/Obs/SDK  |  (pkg/protocol)
                          +--------+---------+
                                   |
         +-------------------------+-------------------------+
         |                         |                         |
         v                         v                         v
+------------------+     +------------------+     +------------------+
|  Policy Engine   |     | Interactive Appr |     |   Trace Logger   |
| (internal/policy)|     |(internal/approv) |     | (internal/trace) |
+--------+---------+     +---------+--------+     +---------+--------+
         |                         |                         |
         +-------------------------+-------------------------+
                                   |
                                   v
                          +------------------+
                          |  Runtime Router  |  (internal/runtime)
                          +--------+---------+
                                   |
                    +--------------+--------------+
                    |                             |
                    v                             v
       +-------------------------+   +-------------------------+
       |   Local Host Runtime    |   |     Docker Runtime      |
       |    (runtimes/local)     |   |    (runtimes/docker)    |
       +------------+------------+   +------------+------------+
                    |                             |
                    +--------------+--------------+
                                   |
                                   v
                          +------------------+
                          |  Filesystem Diff |  (internal/fsdiff)
                          +------------------+
```

## 2. Command Execution Lifecycle

Below is the step-by-step lifecycle of a run request:

```mermaid
sequenceDiagram
    autonumber
    actor CLI as Developer / AI Agent
    participant Main as Main (main.go)
    participant Policy as Policy (internal/policy)
    participant Appr as Approvals (internal/approvals)
    participant FSDiff as Filesystem Diff (internal/fsdiff)
    participant RT as Pluggable Runtime (runtimes/*)
    participant Logger as Trace Logger (internal/trace)

    CLI->>Main: agentsandbox run "cmd" --backend docker --policy policy.yaml
    Main->>Logger: Initialize RunLogger (creates run_YYYYMMDD_.../)
    
    Note over Main: Policy Verification Phase
    Main->>Policy: Evaluate command prefix
    alt Denied rule matched
        Policy-->>Main: Blocked
        Main->>Logger: Record denial
        Main->>CLI: Return Denied Observation & Exit 1
    alt Require Approval rule matched
        Policy-->>Main: Require Approval
        Main->>Appr: Prompt user
        Appr-->>Main: Approved?
        alt Denied by user
            Main->>Logger: Record denial
            Main->>CLI: Return Denied Observation & Exit 1
        end
    end

    Note over Main: Pre-Execution Phase
    Main->>FSDiff: Take pre-run snapshot (files & SHA-256 hashes)
    
    Note over Main: Execution Phase
    Main->>RT: Run(Action)
    RT->>RT: Construct environment, mount workspace
    RT->>RT: Run command, capture stdout/stderr streams
    RT-->>Main: Raw stdout/stderr & exit code

    Note over Main: Post-Execution Phase
    Main->>FSDiff: Take post-run snapshot
    Main->>FSDiff: Diff pre/post snapshots
    FSDiff-->>Main: List of files added, modified, deleted

    Note over Main: Recording & Completion
    Main->>Logger: Write final report, diffs, & raw stdout/stderr logs
    Main-->>CLI: Print Observation (JSON or formatted terminal console)
```

## 3. Directory Structure & Code Responsibilities

* **`cmd/agentsandbox/main.go`**: The orchestrator. Parses command-line inputs, configures logging directories, routes execution to runtimes, runs the policy engine, invokes filesystem diffing, and prints output.
* **`pkg/protocol/`**: The core type definition package. Defines the standard JSON schema for an `Action` (the requested task) and `Observation` (the captured outcome).
* **`internal/policy/`**: Implements word-prefix matching. Evaluates commands against YAML policies to assign an effect (`allow`, `deny`, or `require_approval`).
* **`internal/approvals/`**: Interactive CLI prompts. Asks the developer whether to allow a command to proceed, supporting `--yes` (auto-approve) and `--non-interactive` (auto-deny).
* **`internal/runtime/`**: Defines the unified `Runtime` interface and the registry database that lets different backends plug in dynamically.
* **`runtimes/local/`**: Runs commands natively on the host shell (`/bin/sh` or `cmd.exe`).
* **`runtimes/docker/`**: Runs commands inside an ephemeral container. Mounts the workspace to `/workspace` and destroys the container (`--rm`) as soon as execution concludes.
* **`internal/fsdiff/`**: Scans the workspace directory for file additions, modifications, and deletions. Uses SHA-256 checksums to detect file contents mutations.
* **`internal/trace/`**: Collects trace events and writes untruncated program output, trace timelines (`trace.jsonl`), and change logs to the local `.agentsandbox/runs/` folder.
